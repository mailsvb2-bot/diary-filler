# Медицинский автозаполнитель

**Версия:** `v1.3.18-production-quality-gate`

**UI hotfix:** блок 01 стабилизирован: после выбора первичного документа drop-зона не меняет внешнюю высоту, поэтому кнопки «Нет/Да» больше не должны прыгать.

**Audit hardening:** усилены проверки входных файлов/типов документов/дат, чтение ЭПИ из `.docm` и TXT `cp1251`, fallback drag-and-drop нескольких файлов, карантин неверного `settings.json`; дополнительно закрыты неверные расширения, дата выписки раньше поступления, cp1251 TXT через drag-and-drop, повторные дневниковые статусы и зависание/дублирование release-gate.

**Contract tests:** добавлен пользовательский end-to-end контракт `выбор документа → merged popup → создание выбранного комплекта → проверка содержимого DOCX`, включая сценарий отмены popup без частичного создания файлов.

**Silent success UX:** после успешного создания программа больше не показывает информационные окна подтверждения и confirmation-popup для сценария «только дневники»; вместо этого сохраняет файлы и открывает папку результата.

**Deep service-boundary audit:** усилен прямой слой генерации DOCX: теперь он не допускает пустые ФИО/дату поступления/номер истории/диагноз/лечение, невозможные даты комиссий/ВК раньше поступления и дубли слов «военкомат/военный комиссариат» в итоговых формулировках.

Локальная Windows-программа для заполнения медицинских документов и дневников наблюдения в Word `.docx`.

Главный production-контракт: обычному пользователю отдаётся готовый `MedicalDiaryAutofill.exe`. Python, pip и зависимости нужны только разработчику или сборочному GitHub Actions runner.

## Возможности

- единый Tkinter UI без тяжёлых UI-фреймворков;
- создание выбранных медицинских документов по встроенным шаблонам;
- заполнение дневников наблюдения;
- выбор документов кнопками-плитками: можно создать один документ, несколько или весь комплект;
- выбранные карточки и нажимаемые кнопки получают лёгкий цветной градиент, чтобы выбор был заметен без чёрных галочек;
- сценарии `создать и сохранить без печати` и `создать, сохранить, распечатать`;
- выбор принтера на Windows;
- drag-and-drop зона: можно перетащить первичный осмотр, направление, ЭПИ, тексты, таблицы дневников или папку шаблонов `01–31` прямо в окно;
- кнопки `Тексты` и `Даты` остаются визуально подсвеченными мягким градиентом после выбора соответствующих файлов;
- автоопределение типа первичного документа: `Первичный осмотр` или `Направление на госпитализацию`;
- при направлении на госпитализацию popup запрашивает номер истории болезни, лечение и диагноз;
- при первичном осмотре popup не открывается: данные берутся из самого DOCX;
- парсинг даты поступления из строки/ячейки рядом с названием документа, например `12.01.2026 Первичный осмотр`;
- дата рядом с ФИО/датой рождения не используется как дата поступления;
- парсинг ФИО, возраста/года рождения, адреса и места работы как в столбик, так и в одну строку;
- парсинг клинических блоков, диагноза, лечения, анамнеза жизни и ЭПИ;
- экспертный анамнез из UI: в строке `Нужен больничный лист` по умолчанию стоит `нет`, кнопка `Заполнить` открывает ввод даты начала, организации и должности;
- автоматическая вставка экспертного анамнеза в первичный осмотр, выписной эпикриз и совместный осмотр;
- встроенные шаблоны хранятся в `embedded_templates.py`, поэтому EXE может работать без отдельной папки `templates/`.

## Production-запуск для пользователя

Пользователю нужен только файл:

```text
dist\MedicalDiaryAutofill.exe
```

Он запускается двойным кликом. Устанавливать Python, pip, `python-docx`, `pywin32` или другие зависимости пользователю не нужно.

## Запуск из исходников для разработчика

На Windows:

```bat
start_windows.bat
```

Launcher создаёт локальное окружение `.venv_runtime` и ставит зависимости туда, не трогая глобальный Python.

Диагностический запуск:

```bat
start_windows_debug.bat
```

Запуск без консольного окна:

```bat
start_windows_no_console.vbs
```

## Сборка EXE на Windows

```bat
build_exe_windows.bat
```

Скрипт делает следующее:

1. создаёт `.venv_build`;
2. ставит `requirements_build.txt`;
3. запускает `release_check.py`;
4. собирает `dist\MedicalDiaryAutofill.exe` через PyInstaller;
5. добавляет Windows version metadata из `version_info.txt`;
6. собирает без UPX (`--noupx`), чтобы снизить риск ложных antivirus-срабатываний.

## Release-gate

Перед публикацией исходников или EXE должен проходить:

```bat
python release_check.py
```

Он проверяет:

- наличие обязательных файлов;
- компиляцию всех `.py`;
- `smoke_test.py`;
- `smoke_test_combined.py`;
- контракт безопасного `settings.json`;
- контракт сборочного скрипта и GitHub Actions;
- отсутствие мусора: `__pycache__`, `.pyc`, `build`, `dist`, `test_run*`, временных отчётов.

## Проверка drag-and-drop contract

В исходниках есть отдельная headless-проверка:

```bat
python dnd_contract_check.py
```

Она проверяет, что зона `Перетащите сюда...` зарегистрирована как drop-target, что TkDND runtime-путь не потерян, и что Windows-пути с пробелами/кириллицей корректно разбираются из `<<Drop>>` события. Реальную доставку файлов из Проводника Windows всё равно нужно проверить вручную в EXE перед коммерческим запуском.

## Drag-and-drop сценарий

В блоке `02 Входные файлы` есть зона для перетаскивания файлов.

Что делает программа:

- если перетащить `Направление на госпитализацию`, программа распознаёт файл и открывает popup с полями: `Номер истории болезни`, `Лечение`, `Диагноз`;
- если перетащить `Первичный осмотр`, popup не появляется — данные берутся из DOCX;
- если перетащить ЭПИ, файл подставляется как ЭПИ;
- если перетащить таблицы/тексты дневников, они распределяются по соответствующим полям;
- если библиотека `tkinterdnd2` недоступна, программа всё равно запускается, а drag-and-drop зона работает как большая кнопка выбора файла.

## Автоподбор шаблона дневников 01–31

1. В первичном документе находится дата рядом с названием документа, например `12.01.2026 Первичный осмотр`.
2. Программа ищет шаблон по дню поступления: `12.docx`, `12(...).docx`, `№12.docx` и похожие имена.
3. Если такого файла нет, используется резервная совместимость со старыми наборами: поиск шаблона следующего дня, например `13.docx`.
4. Найденный файл подставляется в поле `Даты`, дальше работает заполнитель дневников.

Папку шаблонов можно задать так:

- перетащить папку `01–31` в drag-and-drop зону;
- выбрать любой файл из этой папки через кнопку `Даты`;
- выбрать папку, когда программа попросит при создании дневников.

## Экспертный анамнез и больничный лист

В модуле `01 Данные пациента` есть пункт `Нужен больничный лист`. По умолчанию в поле стоит `нет`.

Popup открывается только кнопкой `Заполнить`. После успешного заполнения поле переходит в состояние `да`. В popup врач указывает:

- с какого числа нужен больничный лист;
- где работает пациент / организацию;
- должность.

Эти данные не сохраняются между запусками и используются только для текущего комплекта документов.

## Конфиденциальность

Программа не ведёт историю пациентов и не сохраняет в `settings.json` ФИО, диагноз, лечение, даты, выбранные документы или созданные файлы.

В `settings.json` сохраняются только технические удобства:

- последние папки диалогов выбора файлов;
- выбранный принтер.

Если `settings.json` повреждён, программа не падает: битый файл переносится в `settings.broken.<timestamp>.json`, а запуск продолжается с пустыми настройками.

В обычном режиме в папку результата сохраняются только выбранные `.docx`-документы. TXT-отчёты создаются только при явном debug-режиме:

```bat
set MEDICAL_AUTOFILL_WRITE_REPORTS=1
```

## Если программа не запускается

Запустите:

```bat
start_windows_debug.bat
```

Если окно не открылось, рядом с программой появится `startup_error.log` с точной причиной.

## Текущий контракт дневников

- дата поступления берётся из заголовка первичного документа или направления, например `15.04.2026 Направление на госпитализацию`;
- дата рождения рядом с ФИО не используется для дневников;
- папка шаблонов дневников должна содержать DOCX-шаблоны `01.docx–31.docx` или похожие имена (`1.docx`, `№15.docx`, `шаблон 15.docx`);
- кнопка `Даты` показывает DOCX-файлы, но программа сохраняет именно папку и сама выбирает нужный шаблон по дате госпитализации;
- если в таблице шаблона заполнен `День госпитализации`, программа сама заполняет столбцы `Число` и `Месяц/Год` от даты поступления.

## Проверки перед релизом

Перед заливкой на GitHub, сборкой EXE или запуском платного рекламного трафика должны проходить команды:

```bat
python release_check.py
python make_release_zip.py
```

`release_check.py` запускает компиляцию Python-файлов, smoke-тесты, production-контракты и `prod_audit.py`. `prod_audit.py` дополнительно проверяет синхронизацию версии, отсутствие архитектурной пыли после рефакторинга, отсутствие циклических импортов, отсутствие ссылок на удалённые micro-mixin-файлы и наличие release-документов.

## Данные и эксплуатация

Программа работает локально на компьютере пользователя и не отправляет медицинские документы во внешние сервисы. Для продакшн-распространения отдавайте пользователю готовый `MedicalDiaryAutofill.exe`; Python, pip и зависимости пользователю ставить не нужно.

Перед коммерческим запуском на платный трафик обязательно провести ручной Windows QA по `LAUNCH_CHECKLIST.md`: запуск EXE, drag-and-drop, генерация всех документов, печать, проверка антивирусом/Defender SmartScreen и тест на реальных обезличенных DOCX.


## Performance hardening

В v1.3.9-audit-hardening тяжёлые операции вынесены из стартового пути: DOCX/render/ICD/printer/template modules загружаются лениво, первичный DOCX кэшируется по `mtime/size`, поиск принтеров и проверка шаблонов выполняются без блокировки Tk-потока. Перед релизом дополнительно запускается `python performance_check.py`.
